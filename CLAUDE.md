# Claude Code ガイド - D&D 食屍鬼島キャンペーン リプレイ小説プロジェクト

## プロジェクト概要

このプロジェクトは、D&D 5e版とクトゥルフの呼び声TRPGを混ぜたキャンペーン「食屍鬼島」の12回分のセッション記録から、プレイヤーとDMが楽しんで読み返せる「リプレイ小説」を作成するプロジェクトです。

### キャンペーン基本情報

- **DM**: モクモク
- **プレイヤー**:
  - PL1（チョモラン）: 巡礼者
  - PL2（タツヤ）: ノクシル・グリムホルト
  - PL3（まけいぬ）: シャルル・ビアトリス
- **システム**: D&D 5e/2024 + クトゥルフの呼び声TRPG（ごちゃ混ぜルール）
- **セッション数**: 第1回～第12回（継続中）

## 重要：データの信頼性レベル

### ⭐ 一次データ（最優先）

1. **04_プレイヤーログ/** - 各PLが実際にプレイ中・プレイ後に記録したログ
   - 巡礼者ログ: 全12回あり（最も網羅的で信頼性が高い）
   - シャルル日誌/メモ: キャラクター視点の内面描写が豊富（第1,3,5,6,8,12回）
   - ノクシルログ: 第1,5,10回のみだが、内面描写が詳細

2. **07_セッション音声ログ/** - 実際のセッション音声の文字起こし
   - ⚠️ 注意: 雑談も多く含まれる
   - ⚠️ 注意: 録音失敗している回もある
   - 活用方法: 特徴的な台詞や重要な会話の抽出に使用

3. **03_プレイヤー設定/シャルルストーリー/** - DMとPLが作成した公式設定
   - シャルルの過去（ルンジャタ、深きもの）
   - キャラクターの動機や背景の理解に必須

### ⚠️ 二次データ（参考のみ・誤りの可能性あり）

1. **05_リプレイ/** - 他のLLMが生成したリプレイ
   - ⚠️ 誤った解釈や事実誤認が含まれている可能性
   - ⚠️ 一次データと矛盾する場合は一次データを優先
   - 活用方法: 文体やフォーマットの参考程度

2. **08_LLMによるキャラ評/** - LLMが生成したキャラクター分析
   - ⚠️ 公式設定ではない
   - 活用方法: キャラクター理解の補助資料として
   - 📝 **重要**: 各セッション後に更新すること（WORKFLOW.md参照）

## リプレイ小説作成の基本方針

### 1. 情報の優先順位

```
一次データ > 公式設定 > 二次データ（参考）
```

**具体的な優先順位:**
1. プレイヤーログ（04_プレイヤーログ/）
2. 音声ログ（07_セッション音声ログ/）※特徴的な台詞のみ
3. 公式設定（03_プレイヤー設定/）
4. DM公開情報（02_DM公開情報/）
5. 参考資料（05_リプレイ/、08_LLMによるキャラ評/）

### 2. 執筆時の注意点

#### ✅ DO（推奨）

- 一次データ（プレイヤーログ）を最優先で参照する
- 複数のプレイヤーログを照らし合わせて事実確認する
- 音声ログから特徴的な台詞を抽出して臨場感を出す
- シャルルストーリーなどの背景設定を活用して深みを出す
- 各キャラクターの「声」（文体・話し方）を尊重する

#### ❌ DON'T（非推奨）

- 05_リプレイ/ の内容を鵜呑みにしない
- 一次データと矛盾する情報を採用しない
- プレイヤーログにない出来事を創作しない
- 音声ログの雑談部分を本編に混ぜない

### 3. 情報の照合方法

**例: 第7回「血溜まり部屋の戦闘」を執筆する場合**

1. **巡礼者ログ** (`04_プレイヤーログ/巡礼者_PLチョモラン/第7回目.txt`) を読む
   - 時系列の流れを把握
   - 戦闘の経過を確認

2. **シャルル日誌** (`04_プレイヤーログ/シャルル_PLまけいぬ/第X回目_シャルル日誌.txt`) があれば読む
   - シャルルの内面描写を補完
   - 巡礼者ログと矛盾がないか確認

3. **音声ログ** (`07_セッション音声ログ/第7回目-1.srt`) から台詞を抽出
   - 特徴的な会話シーンを探す
   - プレイヤーログで確認できなかった台詞を補完

4. **05_リプレイ/第7回_*.md** は参考程度に
   - 構成やシーン分けのヒントとして活用
   - 内容の正確性は一次データで検証

## よくある質問（FAQ）

### Q1: 05_リプレイ/ のファイルは使えないの？

A: 参考にはなりますが、**誤りが含まれている可能性があるため鵜呑みにしない**でください。
- 文体やフォーマットの参考にする ✅
- シーン分けの参考にする ✅
- 事実関係を一次データで必ず検証する ⚠️

### Q2: 音声ログはどこまで使うべき？

A: **特徴的な台詞や重要な会話のみ**抽出してください。
- キャラクターの特徴的な台詞 ✅
- NPCとの重要な会話 ✅
- 雑談部分 ❌
- プレイヤーログと矛盾する内容 ❌

### Q3: プレイヤーログ同士で矛盾がある場合は？

A: 以下の優先順位で判断してください：
1. **複数のログで一致している情報** → 採用
2. **一方しか記録していない情報** → より詳細な方を採用
3. **明らかに矛盾している情報** → DMに確認、または両方の視点として記載

### Q4: キャラクターの内面描写はどこまで書いていい？

A: プレイヤーログに記載されている範囲内で：
- シャルル日誌の内面描写 → そのまま使用可 ✅
- ノクシルログの内面描写 → そのまま使用可 ✅
- 巡礼者ログの描写 → そのまま使用可 ✅
- ログにない内面描写 → 創作しない ❌

### Q5: 第12回以降の新しい回はどうする？

A: **標準ワークフローに従ってください**

詳細は `05_リプレイ/_ガイドライン/WORKFLOW.md` を参照。

**概要:**
1. データ収集・整理（プレイヤーログ、音声ログ）
2. リプレイ初稿作成（一次データを最優先）
3. 文体統一チェック（文体統一ガイドライン.md）
4. セリフ洗練化（セリフ洗練化ガイドライン.md）
5. **キャラクター評価更新**（08_LLMによるキャラ評/）
6. 最終レビュー・コミット
7. **Webサイトへの反映**（09_Webサイト/）

## 協働型マルチエージェントワークフロー

**重要更新（2026-01-01）:** 第12回以降のリプレイ作成は、原作者・編集者・校正者エージェントが協働する新しいワークフローに移行しました。

### 🎯 ワークフローの概要

**目的:** プロジェクトオーナー、原作者エージェント、編集者エージェント、校正エージェントの協働により、**読み物としての面白さと正確性を両立**した高品質なリプレイ小説を作成する

```
セッション実施
    ↓
【Phase 0】素材準備（プロジェクトオーナー）
    ├─ 時系列メモ作成（粗筋レベル）
    ├─ セリフ自動抽出
    └─ セリフ校正・ユーザー確認
    ↓
【Phase 1】構成設計（原作者 ↔ 編集者）
    ├─ 原作者: 構成案作成
    ├─ 編集者: 構成レビュー（テンポ、Show vs Tell等）
    └─ 原作者: 構成最終調整
    ↓
【Phase 2】初稿執筆（原作者）
    └─ 構成に基づき初稿作成
    ↓
【Phase 3】編集レビュー（編集者 ↔ 原作者）
    ├─ 編集者: 読み物クオリティチェック
    ├─ 原作者: 編集指摘への対応
    └─ ※品質基準未達の場合はPhase 2へ戻る
    ↓
【Phase 4】校正レビュー（校正者 ↔ 原作者）
    ├─ 校正者: 文章の正確性チェック
    ├─ 原作者: 校正指摘への対応
    └─ ※品質基準未達の場合はPhase 2へ戻る
    ↓
【Phase 5】事実確認（検証エージェント ↔ 原作者）
    ├─ 検証エージェント: 一次データとの照合
    ├─ 原作者: 事実誤認の修正
    └─ ※矛盾が多い場合はPhase 2へ戻る
    ↓
【Phase 6】ユーザー最終確認
    └─ プロジェクトオーナーによる承認
    ↓
【Phase 7】キャラクター評価更新
    └─ キャラ評エージェント: PCおよびNPC評価を更新
    ↓
【Phase 8】完成・公開
    └─ GitコミットおよびWebサイトへの反映
    ↓
【Meta-Review】複数回振り返り（3話ごと or 重要分岐点）
    ├─ 全エージェント参加
    ├─ 伏線の一貫性チェック
    ├─ キャラクター描写の一貫性チェック
    └─ ガイドラインのアップデート
```

### 👥 エージェントの役割

1. **プロジェクトオーナー（ユーザー）**
   - 一次データ提供、時系列メモ作成、最終承認

2. **原作者エージェント（Sonnet推奨）**
   - リプレイ小説の執筆、構成設計、編集・校正指摘への対応

3. **編集者エージェント（Sonnet推奨）**
   - 読み物としてのクオリティチェック（Show vs Tell、テンポ、冗長性など）
   - 参照: [編集ガイドライン.md](05_リプレイ/_ガイドライン/編集ガイドライン.md)

4. **校正者エージェント（Haiku推奨）**
   - 文章の正確性チェック（文法、句読点、表記ゆれ、誤字脱字）
   - 参照: [校正ガイドライン.md](05_リプレイ/_ガイドライン/校正ガイドライン.md)

5. **検証エージェント（Haiku推奨）**
   - 一次データとの事実照合、時系列チェック

6. **キャラ評エージェント（Sonnet推奨）**
   - PCおよびNPCのキャラクター評価更新

### 📚 詳細なワークフロー

**協働型マルチエージェントワークフローの詳細は、以下のドキュメントを参照してください:**

- **[WORKFLOW.md](05_リプレイ/_ガイドライン/WORKFLOW.md)** - 全体のワークフロー、各Phaseの詳細手順、品質ゲート基準
- **[編集ガイドライン.md](05_リプレイ/_ガイドライン/編集ガイドライン.md)** - Show vs Tell、テンポ、冗長性、クリフハンガーなど
- **[校正ガイドライン.md](05_リプレイ/_ガイドライン/校正ガイドライン.md)** - 文法、句読点、表記ゆれ、誤字脱字など
- **[文体統一ガイドライン.md](05_リプレイ/_ガイドライン/文体統一ガイドライン.md)** - 地の文、主語、短文・長文のバランス
- **[セリフ洗練化ガイドライン.md](05_リプレイ/_ガイドライン/セリフ洗練化ガイドライン.md)** - 音声ログ誤変換修正、フィラー削除、キャラクター別口調

### 🔑 重要な変更点

**旧ワークフロー（第1～11回）との違い:**

1. **エージェント間の協働:** 原作者・編集者・校正者が互いにレビュー、ダメ出し、代案を出し合う
2. **品質ゲート:** 各Phaseで品質基準を設け、未達の場合は前のPhaseに戻る
3. **読み物クオリティの重視:** 編集者エージェントが読者視点でレビュー（Show vs Tell、テンポなど）
4. **文章の正確性担保:** 校正者エージェントが機械的にエラーをチェック
5. **Meta-Review:** 3話ごとに複数回を振り返り、伏線やキャラクター描写の一貫性を確認

### 🎯 品質ゲート基準まとめ

| Phase | 担当エージェント | 合格基準 | 不合格時の対応 |
|---|---|---|---|
| **Phase 1** | 原作者 ↔ 編集者 | 編集者評価B以上 | 構成案を修正（v2作成） |
| **Phase 3** | 編集者 | 総合評価B以上 | Phase 2へ戻って再執筆（C評価の場合） |
| **Phase 4** | 校正者 | 誤り15個以下 | Phase 2へ戻って再執筆（16個以上） |
| **Phase 5** | 検証者 | 矛盾3個以下 | Phase 2へ戻って再執筆（4個以上） |
| **Phase 6** | ユーザー | ユーザー承認 | 該当Phaseへ戻る |

## 執筆時の文体・スタイル指針

### 基本方針
- 読み物として面白い
- 内輪の雰囲気も大切にする
- 各キャラクターの「声」を尊重

### キャラクター別の文体

**巡礼者**
- 異形だが純粋
- 人間の「矜持」や「文化」に興味を持つ
- 直接的で率直な物言い

**ノクシル**
- 寡黙でクール
- 「問題ない」が口癖
- レンジャーとしての観察眼

**シャルル**
- パッション溢れる
- 正義感が強い
- 内面の葛藤も描写

### 描写のバランス
- 戦闘: 臨場感重視、重要な判定を詳細に
- 会話: 特徴的な台詞を活かす
- 内面: プレイヤーログの記述を尊重
- 伏線: 次回への引きを明確に

## Webサイトデプロイ手順

このプロジェクトでは、MkDocsを使用してGitHub Pagesで静的サイトを公開しています。

### ⚙️ 自動生成ファイルの仕組み

**重要:** 以下のファイルは `.gitignore` で除外され、**自動生成される**ため、手動で編集しないでください。

```
09_Webサイト/docs/index.md           # 05_リプレイ/_Webコンテンツ/ からコピー
09_Webサイト/docs/world.md           # 05_リプレイ/_Webコンテンツ/ からコピー
09_Webサイト/docs/replays/*.md       # 05_リプレイ/ からコピー（session01.md - session11.md形式）
09_Webサイト/docs/characters/*.md    # 08_LLMによるキャラ評/ からコピー（画像パス自動修正）
```

### 📋 デプロイ手順（第12回以降の新規セッション追加時）

#### Step 1: リプレイファイルをWebサイトにコピー


#### Step 2: キャラクター評価ファイルを更新（必要に応じて）

新しいNPCのイラストがある場合：

```yaml
    - 第12回 （タイトル）: replays/session12.md  # 新規追加
```

`05_リプレイ/_Webコンテンツ/index.md` を編集し、新しいセッションを追加：

### 📝 よくあるトラブルシューティング

#### 1. CSSが崩れている

**原因:** `site_url` が mkdocs.yml に設定されていない

**解決策:**
```yaml
site_url: https://Makeinu1.github.io/DaD_Ghoul_Island/
```

#### 2. 404エラーが発生する

**原因:** mkdocs.yml の nav セクションとファイルパスが一致していない

**解決策:** navセクションのパスとファイルパスを確認

#### 3. 画像が表示されない

**原因:** 画像パスが間違っている（`../06_イラスト/` のまま）

**解決策:**
```bash
cd "09_Webサイト/docs/characters"
sed -i '' 's|../06_イラスト/|../images/|g' *.md
```

#### 4. ビルド時に警告が出る

**原因:** リンク先のファイルが存在しない、またはアンカーが間違っている

**解決策:** 警告メッセージを確認し、該当ファイルを修正
